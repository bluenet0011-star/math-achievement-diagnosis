# 조작형 활동 저작 가이드 (틀) — 고정 규격

> 모든 활동은 이 틀을 따른다. **"이 조작을 해내면 그 수행능력을 보유한 것으로 인정한다"**는 평가관이 핵심이다.
> 단순 문제풀이가 아니라 **학생이 직접 조작·구성**하게 하고, **한 조작이 여러 수행능력을 동시에 입증**할 수 있다.

---

## 0. 저작 철학 (항상 점검)

1. **증거 기반(evidence-of-skill)**: 각 조작(interaction)은 "이걸 성공하면 이 수행능력이 있다고 볼 수 있는가?"를 통과해야 한다.
2. **조작·구성 중심**: 고르기/담기/배치/시뮬레이션 등 **직접 만지는** 활동. 객관식·단답은 보조로만.
3. **다중 수행능력**: 한 조작의 `skillIds`에 여러 개를 넣어 동시에 평가 가능(다른 성취기준 수행능력도 가능).
4. **맥락·이야기**: 도입(`context.story`)에 역사적 사실·실생활·재미있는 일화를 넣어 동기를 만든다.
5. **교육적 의미 검토**: 아래 §5 체크리스트를 매 활동마다 통과시킨다.

---

## 1. 활동 JSON 스키마 (templateType: "interactive")

```json
{
  "id": "act-<코드>",
  "standardId": "<대표 성취기준 코드>",
  "templateType": "interactive",
  "title": "<활동 제목 — 동사형, 직접 해보는 느낌>",
  "context": {
    "story": "<역사·실생활·일화 도입 (2~4문장)>",
    "goal": "<이 활동에서 학생이 할 일 한 줄>"
  },
  "interactions": [
    {
      "id": "ix1",
      "kind": "build-set",
      "prompt": "<무엇을 조작할지 지시문>",
      "skillIds": ["<수행능력 id>", "<여러 개 가능>"],
      "config": { ... },              // kind별 규격(§2)
      "prereq": ["<선수 성취기준 코드>"],  // (선택) 이 조작만 틀렸을 때 역추적 좁히기
      "explain": "<사후 해설 한 줄 — 리포트에서만 공개>"
    }
  ]
}
```

- `skillIds`는 해당 성취기준(또는 다른 성취기준)의 `skills[]`에 **이미 등록된 id**여야 한다.
- 분수는 텍스트로 `1/6`처럼 쓰면 화면에서 **자동으로 위/아래 분수**로 렌더된다(슬래시 금지 규칙 자동 충족).
- 옛 `stepwise`(steps[] + 단일 skillId) 형식도 계속 동작하지만, **신규는 interactive로 작성**한다.

## 2. 조작 부품 라이브러리 (현재 구현)

각 부품은 `render(ix,host,act)` + `grade(ix,act)→true/false` 계약을 지킨다. 새 부품은 `js/templates.js`의 `Kinds`에 키 하나만 추가하면 엔진 수정 없이 붙는다.

### `build-set` — 칩을 골라 집합/사건/값의 모임을 직접 구성
```json
"config": {
  "basketName": "S",
  "items": [{"label":"1","val":1}, {"label":"2","val":2}],
  "target": [1,2,3,4,5,6]
}
```
- 학생이 칩을 눌러 담는다. **담은 값들의 집합 == target** 이면 통과.
- 용도: 표본공간·사건 구성, 옳은 설명 고르기(label에 문장), 확률값 가능값 고르기 등.

### `compute` — 정의·공식을 적용해 값 입력 (정수)
```json
"config": { "prefix":"P(A) =", "denominator":6, "answer":2 }     // 분수형: 분자 입력 / 분모 6
"config": { "prefix":"경우의 수 =", "suffix":"가지", "answer":8 } // 일반형
```
- `answer`와 일치하면 통과. `denominator`를 주면 입력칸이 **분자**가 되어 위/아래 분수로 표시.

### `simulate` — 직접 반복 시행 → 상대도수가 이론값에 수렴함을 관찰 → 이론값 선택
```json
"config": {
  "trials": 60, "faces": 6, "success": [3,6],
  "question": "이 사건의 수학적(이론적) 확률은?",
  "choices": ["1/6","1/3","1/2"], "answer": 1
}
```
- 실제 주사위가 굴러가고 상대도수 추이 그래프가 그려진다(큰수의 법칙 체감). 충분히 굴린 뒤 이론값을 고르면 통과.

### `expr` — 식을 입력 (간이 수식 입력기 + 대수적 동치 채점)
```json
"config": { "answer":"x^2+2x+1", "accept":["(x+1)^2"], "palette":["x","^","(",")","+","−","×","÷"], "placeholder":"식 입력" }
```
- 학생이 식을 타이핑(기호 버튼으로 보조). 채점은 **대수적 동치**: `x^2+2x+1` 과 `(x+1)^2`, `2x+3` 과 `3+2x` 처럼 **형태가 달라도 같은 식이면 정답**(math.js 수치동치). `2x`, `2(x+1)` 같은 암시적 곱도 인식.

## 2-1. 정답 입력·채점 규칙 (여러 형태 허용)

- **수치·분수(`compute`)**: `1/2`, `0.5`, `2/4` 를 모두 같은 값으로 인정(수치 동치). `answer`는 대표값 하나만 적으면 됨. (분수 표시가 필요하면 `denominator`로 분자 입력형)
- **식(`expr`)**: `answer`에 한 가지 형태만 적어도 동치인 모든 형태를 정답 처리. 특정 형태만 받고 싶으면 `accept`에 한정 목록.
- **문자열 다답**: `compute`/`fill-table`에 `accept`(허용 목록)를 주면 공백·대소문자 무시하고 그 중 하나면 정답.
- **표(`fill-table`)**: 셀 답이 분수/소수면 수치 동치로 비교. 셀별 다답은 `accept:{"r-c":[...]}`.
- 원칙: **"값/식이 같으면 정답"** — 단순 문자열 일치가 아니라 의미 동치로 채점.

### `slider` — 매개변수를 조절해 그래프/값을 목표에 맞춰 **구성**
```json
"config": { "preview":{"expr":"a*sin(b*x)","x":[-6.5,6.5],"y":[-3.5,3.5]},
  "sliders":[{"key":"a","label":"a","min":1,"max":3,"step":1,"init":1,"target":2},{"key":"b","min":1,"max":4,"step":1,"init":1,"target":2}], "tolerance":0 }
```
- 슬라이더로 a,b…를 움직이면 **현재 곡선(실선)**이 그려지고 **목표 곡선(점선)**에 맞추는 방식. 각 슬라이더가 `target`에 닿으면 정답. "진폭·주기·평행이동" 등 매개변수의 의미를 직접 조작으로 익힘. (preview.expr은 math.js 식, 변수 x + 슬라이더 key)

### `grid-plot` — 좌표평면에 점을 직접 찍어 그래프/수열/해를 **구성**
```json
"config": { "x":[0,6], "y":[0,15], "target":[[1,2],[2,5],[3,8],[4,11],[5,14]] }
```
- 격자점을 눌러(토글) 점을 찍는다. 찍은 점 집합 == target이면 정답. 수열의 항 (n,aₙ), 함수의 정수점 (x,y), 부등식 해 등을 **직접 그려** 구성.

## 2-2. ★구성형 우선 원칙 (단순 답 구하기 탈피)

활동은 가능하면 **"답을 계산해 입력"을 넘어 학생이 직접 만들고 조작해 구성"**하도록 설계한다.
- "주기를 구하라(compute)" → **"슬라이더로 그 주기의 그래프를 만들어라(slider)"**
- "aₙ을 구하라" → **"항을 좌표에 찍어 직선을 만들어라(grid-plot)"** / "타일로 일반항 식을 조립하라"
- "전개하라" → **"전개식의 항을 골라 담아라(build-set)"** / "단계를 순서대로 세워라(order-seq)"
- 단순 연산 확인이 꼭 필요한 부분은 compute로 두되, **활동마다 최소 1개는 구성형(slider·grid-plot·build-set·order-seq·expr 조립) 조작**을 포함한다.

### 확장 예정 부품(필요 시 추가) — 같은 계약만 지키면 됨
`pascal`(파스칼 삼각형) · `drag-match`(짝짓기) · `place-point`(좌표/수직선 배치) · `slider-explore`(매개변수 슬라이더) · `tree-count`(경우의 수 나무) · `fill-table`(표 채우기) 등. **새 부품을 추가하면 이 문서에도 config 규격을 적는다.**

## 3. 수행능력 선수관계 (활동과 별개의 데이터)

선수관계의 출처는 **수행능력**이다: 수행능력 A가 다른 성취기준의 수행능력 B를 선수로 가지면, 두 성취기준 사이 화살표가 **자동 파생**된다(관리자 → 계통도 편집 → 세부 수행능력 탭 → 🔗선수).
- 활동 저작 시, 어떤 수행능력에 **명확한 선행 능력**이 있으면 관리자에서 선수관계를 함께 등록한다.
- 활동의 수행능력 연결창 상단에는 연결한 수행능력의 **선수 수행능력 추천**이 떠서 함께 연결하기 쉽다.

---

## 4. 저작 절차 (한 활동당)

1. **조사·학습**: 그 개념의 좋은 활동(NCTM Illuminations, GeoGebra, Desmos, 교과서 탐구활동)과 **역사/일화**를 찾아 학습한다.
2. **조작 설계**: 개념의 본질을 드러내는 조작 단위(interactions)를 2~6개로 나눈다.
3. **수행능력 매핑**: 각 조작에 `skillIds`를 단다. 해당 성취기준의 **모든 세부 수행능력이 최소 1개 조작에 연결**되게 한다(커버리지).
4. **선수관계 등록**: 필요한 수행능력 선수관계를 관리자에서 연결.
5. **교육적 의미 점검**(§5) → **분수·접근성** 확인.
6. **검증**: `python3 validate.py` (끊긴 간선·누락 skill·순환 0건), 미리보기로 실제 동작 확인.

## 5. 교육적 의미 체크리스트 (필수 — 매 활동)

- [ ] **개념의 본질**을 조작으로 드러내는가? (정의·구조를 직접 만들어 보는가)
- [ ] **단순 암기·기계적 계산**으로 통과되지 않는가?
- [ ] **오개념을 유발/강화**하지 않는가? (예: 확률값 범위, 독립↔배반 혼동)
- [ ] **맥락/역사/이야기**가 있는가? (동기 부여)
- [ ] **모든 세부 수행능력**이 ≥1 조작에 연결되었는가? (관리자 활동편집의 ✓완전 확인)
- [ ] **접근성**: 분수 위/아래 표기, 색에만 의존하지 않는 표시, 명확한 지시문.
- [ ] 채점이 **결정적**인가? (무작위 탐구라도 채점 기준은 이론값/정답 집합으로 고정)

---

## 6. 저장 위치 / 등록

- 활동: `data/activities/<과목>.json` 의 `"activities"` 배열.
- 수행능력: `data/nodes/<과목>.json` 의 성취기준 `skills`.
- 새 파일은 `data/manifest.json` 에 경로만 추가(코드 수정 불필요).
- 관리자에서 편집한 활동연결·선수관계는 **내보내기**로 받아 위 파일에 반영한다.
